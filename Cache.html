Blocking Cache

The report attached below describes a study on processor performance and memory latency. The primary objective of the study is to investigate the impact of cache memory on processor performance, particularly with regard to memory access latency. I will discuss the details of the design and synthesis of two cache architectures - a direct-mapped cache and a two-way set-associative cache - each with a capacity of 256B and then evaluate the performance of each design using simulation testing.

The baseline design of the cache is a 256B direct-mapped, write-back, write-allocate cache. It is split into control, datapath, and top-level modules to enforce modularity, hierarchy, and encapsulation design principles. The control module is managed by a finite state machine (FSM) controller, which handles states with one or more steps per state during which control signals are sent out to the datapath and status signals are received in return. Included below are state machine and datapath diagrams for the baseline design, which show the different paths taken by the cache for various transactions.

The report also describes the interface for the direct-mapped cache, which consists of various inputs and outputs. The inputs include the clock and reset signals, the test source/cache request message and valid signal, cache response ready signal (from the test sink/processor), memory request ready signal, and the memory response message and valid signal. The outputs include the cache request ready signal, cache response message and valid signal, memory request message and valid signal, as well as the memory response ready signal.

I first fully creating the datapath and then added the different paths: init transaction, read hit path, write hit path, refill path, and evict path. The cache is write-back, write-allocate, which means it writes only to the cache on a write hit and loads the cache line from memory (and then writes to the cache) on a write miss. Additionally, I maintained two register files in the control module for keeping track of valid and dirty cache lines, which are necessary for determining whether evictions are required on cache read/write misses.

I then describe the alternative design of the cache, which is a 256B two-way write-back, write-allocate set-associative cache. The alternative design is also split into control, datapath, and top-level modules, with control being managed by a FSM controller. The alternative design has a least recently used (LRU) replacement policy for choosing which cache lines to replace on read/write misses. Included below are the state machine and datapath diagrams for the alternative design, which show the modifications made to the baseline design to create the alternative design, such as duplicating the tag array and tag check logic, having different valid, dirty, and count (for LRU) register files for each way, and implementing logic to determine which way to write to and to implement the LRU replacement policy. I also tried to improve the modularity of their design by refactoring the tag/data arrays and output logic into separate “way modules.”

Finally, I evaluate the performance of the baseline and alternative cache designs using simulation testing. The simulator harness was built and run using given datasets, as well as datasets added, to generate the total number of cycles and the average number of cycles per transaction for each design. Finally, I conclude that I expect an improvement in the average memory access latency for the two-way set-associative cache design over the direct-mapped cache design, due to the reduced conflict misses with the set-associative cache.
